- **M1（Week 1-2）**：稳定性基线（P0）

## 3. 分阶段开发计划

## P0（Week 1-2）稳定性基线（必须先完成）

### P0-1 迁移治理标准化
**目标**：杜绝“代码已使用字段，线上缺列”问题。  
**任务**：
1. 建立 `schema_migrations` 表。
2. 约定迁移命名：`d1/migrations/YYYYMMDDHHmm__desc.sql`。
3. 增加 migration runner（支持本地/CI/生产）。
4. 在 CI 增加 drift check（关键表关键列）。
5. 发布失败时阻断上线并给出可读原因。

**验收**：
- 每次部署输出当前 migration 版本。
- 任意新字段都通过 migration 发布。

---

### P0-2 认证与关键 API 可观测性
**目标**：将“报错不可定位”改为“5 分钟内定位问题类别”。  
**任务**：
1. 统一错误码（`AUTH_*`、`DB_*`、`VALIDATION_*`）。
2. 登录失败页增加可执行引导（不仅显示错误码）。
3. 关键 API（auth/register/search/post save）增加结构化日志。
4. 引入 request-id，串联前后端日志。

**验收**：
- 登录失败可按错误码直接定位。
- 关键 API 日志具备 request-id、user-id、path、耗时。

---

### P0-3 CI/CD 加固
**目标**：消除“偶发环境问题”。  
**任务**：
1. 固定 Node/pnpm 版本并严格 lockfile。
2. 部署后 smoke：`/`、`/login`、`/api/auth/signin/google`。
3. 失败自动输出上下文（版本、关键 env 检查项、日志链接）。

**验收**：
- 连续 5 次流水线通过率 100%。

---

## P1（Week 3-6）内容生产与搜索能力

### P1-1 编辑器体验提升
**任务**：
1. 自动草稿（按 user+post 维度）。
2. 最近 N 个版本快照（最少 5 版）。
3. 发布前检查清单（标题/摘要/封面/标签/SEO）。

**验收**：
- 从创建到发布的平均操作步数下降 30%。

---

### P1-2 搜索能力升级（轻量）
**任务**：

2. 结果关键词高亮。
3. 增加匿名热词统计。
4. 搜索页分页与缓存策略。

**验收**：
- 搜索接口 P95 < 300ms。

---

### P1-3 SEO 运营化
**任务**：
1. sitemap 覆盖 post/tag/series。
2. canonical 规则稽核。
3. OG 图默认模板与回退策略。
4. 结构化数据字段完善与校验。

**验收**：
- 关键页面 SEO 配置完整且可自动验证。

---

## P2（Week 7-10）后台治理与安全

**验收**：

---

### P2-2 安全基线

**验收**：

## 5. 开发分工建议（最小团队）

- **Owner A（平台/后端）**：迁移、CI、数据层、日志
- **Owner B（业务/前端）**：编辑器、搜索页、管理台交互
- **Owner C（运维/安全）**：限流、告警、secrets 轮换

---

## 6. 风险矩阵与应对

1. **迁移遗漏导致线上报错**  
   应对：部署前强制 migration check，失败阻断。

2. **OAuth 配置变更导致登录中断**  
   应对：保留回滚、错误指引页、灰度验证。

3. **CI 环境波动导致不可复现**  
   应对：固定工具链、锁版本、输出关键上下文。

4. **功能堆叠导致迭代失控**  
   应对：按阶段冻结范围（P0 完成前不接 P2）。
